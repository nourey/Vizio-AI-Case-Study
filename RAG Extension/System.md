# System Design


## What data I would include in RAG

For this system, I would build a hybrid from the start. Because recruiter use cases require two things simultaneously: semantic similarity and strict business rules. For example; for the system to be able to say "bring candidates with Python" semantic retrieval is necessary; but at the same time, to be able to say must be in Berlin, have EU work authorization a very robust structured filter layer is needed. Therefore, I would say that relational DB and vector index should work together. For the data, I would not use only resumes. I would include candidate CV's, structured candidate profiles, skills, work experience, projects, education, recruiter notes, interview feedback, and outreach history. I would also include job descriptions, job requirements, company information, and internal hiring rules. I do this because one CV alone is not enough. A candidate may look strong on paper but maybe salary is too high, maybe the person does not want to relocate or maybe a recruiter already wrote a note saying the candidate is not interested now.

## How I would structure and index it

I would store the data in two parts. First I would use a normal database for structured data. This means fields like location, seniority, salary expectation, work authorization, job status, and pipeline stage. This part is important because recruiters often ask hard filter questions, like show me senior candidates in Berlin who can work remotely. A normal database is best for this kind of filtering. Furthermore, I would use a vector database for text search. This is for resume text, job descriptions, recruiter notes, interview summaries, and company notes. But I would not put everything into one big index. I would separate them. For example; I would have candidate chunks, job chunks, company chunks, and policy chunks. I do this because different questions need different data. If everything is mixed together, search results can become noisy and less accurate.

For indexing I would not cut every document into random same size pieces only. I would split documents in a meaningful way. In a CV I would keep summary, each work experience, each project, and each note as separate chunks. In a job description, I would separate role summary, responsibilities, required skills, and nice to have skills. This helps the system find more exact information. Then the AI can say things like: “This candidate matches because of Airflow and TimescaleDB experience” instead of giving gibberish answers.

On the embedding model side, I would not just choose one model and embed everything with it. My first decision would be based on the text type and language. If the system mostly consists of English CV's and job descriptions, I would start with one of the strong text embedding models and my first goal would be semantic matching quality. But in the recruiter use case, general semantic similarity alone is not enough: skill, title, tool and domain terms need to be well captured. Therefore, I would choose the embedding model by benchmarking. For example I would have a small evolution set: truly suitable candidate chunks for specific jobs, unsuitable chunks and borderline examples. I would try different embedding models in the same retrieval pipeline and compare them using metrics like recall and precision after rerank. The goal here would not be to find the coolest model but the model that produces the least noise in my data type. In some areas, instead of embedding raw text, I would consider generating normalized representations first. For example, instead of directly using raw skill descriptions from a candidate's CV I would map them to a canonical skill taxonomy, storing them in a structured format and adding them as supporting information for embedding. For instance phrases like workflow orchestration with Airflow, Airflow DAG development, and Apache Airflow, while spelled differently, can be linked to the same set as canonical skills. This significantly improves retrieval quality. I would do the same for title normalization. Because roles like Data Engineer, Analytics Engineer, Platform Data Engineer, or ETL Developer are sometimes semantically similar but not exactly the same. Keeping these as normalized fields is very useful in both filtering and reranking.

When creating embeddings I would not just embed the bare text and leave it at that. Including title and context information in some chunks improves retrieval. For example when embedding a candidate project chunk I could subtly incorporate not only the project description but also that it is a candidate project, the candidate's title, and the role context at that time. Similarly when embedding recruiter notes, it is important to include that it is a note, the date, and the relevant role context. Of course, the key here is to avoid cluttering the embedding with unnecessary metadata. But if the correct context is added, the retrieval quality visibly improves.

When choosing a vector database, I would decide based on the usage pattern. If the system will not receive very high write traffic but will receive frequent queries, a classic ANN based vector store would be perfectly sufficient. However, I would optimize the approximate nearest neighbor index settings according to the data size. Aggressive ANN settings might be unnecessary for very small datasets for large datasets, the balance between recall and latency becomes important. Also I would not just use dense retrieval on the candidate and job sides. If possible I would use hybrid search that is, I would combine dense vector similarity with lexical signals. Because sometimes certain terms are very critical in recruiter queries. For example, words like dbt, Snowflake, Kubernetes, and German language should not be confused with semantically similar terms. Hybrid retrieval is therefore very useful.

I would think of the retrieval process in three layers. The first layer would be a structured pre filter. At this stage, the candidate pool is narrowed down using strict business criteria. The second layer would be semantic retrieval here, suitable candidate chunks, projects, notes, or similar past placements are extracted. The third layer would be reranking. I believe reranking is crucial in recruiter use cases because embedding similarity often finds relevant but not most suitable. In the reranking phase, I would use a combination of must have skill coverage, title closeness, experience recency, domain fit, and some structured match features. If necessary, a more expensive but higher quality model like cross encoding can also be implemented here. This is especially valuable when narrowing down the top 50 results to 10.

## Metadata and access control

The metadata side is important not only for access control but also for retrieval quality. For each chunk, I would maintain fields like entity_type, chunk_type, entity_id, tenant_id, team_id, visibility_scope, sensitivity_level, created_at, updated_at, embedding_version, and parser_version. Additionally it would be beneficial to move some role or candidate related normalized fields to the metadata filter. For example fields like seniority_normalized, location_normalized, and skills_normalized could help with pre retrieval filtering or candidate pool narrowing. This way, I would not burden the entire process with embedding similarity.

Regarding access control, my clearest stance would be retrieval time enforcement. Unauthorized data should not enter the model context. To achieve this, I would use metadata filtering on the vector index side. For example, recruiter note chunks should only be retrievable by users with the relevant team or role. The same logic would apply to salary expectations or sensitive interview feedback. It would not be enough to just check these fields in the relational DB, the same rule should apply on the vector retrieval side. Otherwise the model would technically see the unauthorized data.

## Update strategy

In my update strategy, I would not leave the retrieval system static. When a new CV arrives, a recruiter note is added, or the job description changes, instead of performing a full reindex, I would apply a partial reindex at the affected entity level. For example, if a candidates new note arrives, only the relevant note chunk and, if necessary, the candidate summary chunk are regenerated. If the parser changes or the embedding model is upgraded, then a full reindex path is required. That is why versioning is so important. I would keep a document_version, parser_version, and embedding_version in each chunk. This way, when retrieval quality decreases, I can answer questions like is the data outdated, is the parser broken, are the embeddings heterogeneous?

On the DB tuning side, especially in the candidate job matching product, I would prioritize two things: query planning and materialized helper structures. For example; instead of recalculating frequently used derived features in every query, it might be logical to store them in pre calculated summary tables or materialized views. For example, fields like the candidates normalized primary skills list, last updated date, last active pipeline stage, or job side must have skill set can be pre calculated. This speeds up the retrieval pipeline and reduces unnecessary repetition on the implementation side.

## Failure modes

Finally I would manage the quality of this system not just by checking if it works, but through retrieval evaluation. That is, I would create a job to candidate retrieval test set. I would measure which candidate chunks should rank higher for specific roles, which recruiter notes are useful, and which retrieval combinations generate noise. Embedding model selection, chunk size, hybrid weights, reranking strategy, and metadata filters would be optimized based on this evaluation set. Because this is where RAG systems are most likely to excel or fail the pipeline may seem logical on paper but if the actual retrieval quality is not tested, the system will produce irrelevant but convincing answers.
